Experiments

trackpad-lag fixes / sandbox
Each version has a colored dot over the TV so you can confirm which one you're on.
All 4 share the same baseline: audit fixes (raycast/inktrail skipped during scroll, CRT video throttled, bloom+outline fade, OS momentum-tail kill).
Each adds ONE distinct experimental change on top.
v1 — Audit-fixed baseline
All 5 audit findings + the bloom+outline fade + momentum-tail kill applied.
Real bloom/outline fade (was missing in prior v1). updateHoverGlow + renderInkTrail skipped during scroll. CRT VideoTexture throttled to every-other-frame during scroll. OS momentum-tail snap. This is the proper baseline to compare against.
v2a — Render-on-demand
Skip composer.render() entirely when nothing changed in last 500ms.
Idle viewing (reading, paused on a section) → GPU completely rests, no render at all. Only renders when: user input within 500ms, bloom fade still lerping, or button animation in flight. Three.js fundamentals pattern. R3F uses frameloop="demand" for this.
v2b — 30fps during scroll
Skip every-other composer.render() while scrolling — game-cutscene trick.
During scroll-active window, only render on even frames (effectively 30fps cinematic). Cuts render cost ~50% during the heaviest moment. Camera motion still tracks scroll, just refreshed at 30Hz. Returns to 60fps when idle.
v2c — CSS scroll-driven (experimental)
Move scroll math from main thread to compositor via animation-timeline: scroll().
Uses CSS @property + animation-timeline (Chrome 115+, Safari 18+, Firefox flag-gated 2026). The browser computes scroll progress on the compositor thread; JS reads the resolved value once per rAF. Falls back to window.scrollY on unsupported browsers. Most experimental of the four.
